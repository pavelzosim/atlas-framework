01 Introduction — ShaderLab vs CG/HLSL

This is a structured, production-oriented reference for technical artists working with hand-written shaders in Unity. It covers shader code structure, property declaration, GPU execution flow, and pipeline-specific differences across BiRP, URP, and HDRP.

Δ1 // CG_HLSL_PROGRAM — Core
Low-level GPU execution language. Shader code lives inside CGPROGRAM / ENDCG (legacy) or HLSLPROGRAM / ENDHLSL (modern SRP) blocks. Compiled directly to GPU bytecode.
Δ2 // SHADERLAB — Wrapper
Unity-specific declarative layer. Defines properties, SubShaders, passes, and fallback states. Orchestrates how your HLSL code connects to the Material Inspector UI and the rendering pipeline.

Timeline & Pipeline Status (2024–2026)

Strategy shift (2026): Unity has officially consolidated all graphics engineering into URP. HDRP enters maintenance mode. BiRP deprecation begins in Unity 6.5 — supported through 6.7 LTS (~2028) for Enterprise, then EOL.

[ 2024 ] Legacy Cleansing & SRP Convergence

  • CGPROGRAM fully deprecated in URP/HDRP. HLSLPROGRAM / ENDHLSL is mandatory for modern shader target compatibility.
  • URP 3D Textures (v12+): Stable native support for volumetric effects within Universal pipeline.
  • Render Graph Architecture: Initial framework convergence aligning URP and HDRP execution passes under the hood.

[ 2025 ] Unity 6 Production Lifecycle

  • GPU Resident Drawer & Render Graph: Massive draw-call reduction via hardware-driven data residency and automatic frame-graph optimization.
  • Unity 6.1 — Deferred+ & VRS: Deferred+ rendering path in URP + native Variable Rate Shading for modern hardware.
  • Advanced Feature Reflow: Motion Vectors, Enhanced Alpha Processing, Lights 2D upgrades — URP approaching old HDRP feature parity.

[ 2026 ] The Single Architecture Push

  • URP as master pipeline — receives Physical Light Units, Auto-Exposure, Physical Sky, SSR, and real-time GI previously locked to HDRP.
  • HDRP frozen — stability, bug fixes, Nintendo Switch 2 preview only.
  • BiRP EOL begins Unity 6.5 — all Asset Store assets and docs transition to URP defaults.

02 GPU Execution Pipeline

The GPU reads shader code sequentially, top to bottom. The execution pipeline inside a CGPROGRAM / HLSLPROGRAM block follows a strict stage sequence:

1
Properties
Material Inspector parameters — textures, vectors, colors, floats.
2
SubShader Setup
Hardware tags, render states, LOD, compilation passes.
3
Variables & Structures
Global uniforms, CBUFFER blocks, Attributes / Varyings / appdata / v2f.
4
Helper Functions
Math utilities, lighting functions — must be declared before use (forward eval only).
5
Vertex Shader
Per-vertex: object-space → clip-space geometry transformation.
6
Rasterization
Fixed-function hardware stage: primitives → pixel fragments. No sort guarantee.
7
Fragment Shader
Per-pixel: texturing, lighting, final color. Massive parallel execution — no ordering.
8
Frame Buffer
Output: alpha blending, depth/stencil testing, raster presentation.

Execution rules: Functions must be declared before they are called. The Vertex stage always completes before the Fragment stage. The hardware processes thousands of vertices and pixels simultaneously — there is zero ordering guarantee within the same stage.

Basic Color Shader — Full Blueprint (URP)

Production-ready unlit shader demonstrating how ShaderLab properties bind to uniform variables inside the HLSL stages:

Shader "Custom/Pipeline/BasicColor"
{
    Properties
    {
        _Color ("Main Color", Color) = (1.0, 1.0, 1.0, 1.0)
    }

    SubShader
    {
        Tags
        {
            "RenderType"     = "Opaque"
            "RenderPipeline" = "UniversalPipeline"
        }

        Pass
        {
            Name "ForwardLit"
            HLSLPROGRAM
            #pragma vertex vert
            #pragma fragment frag

            #include "Packages/com.unity.render-pipelines.universal/ShaderLibrary/Core.hlsl"

            // CPU → Vertex Shader
            struct Attributes { float4 positionOS : POSITION; };
            // Vertex Shader → Fragment Shader
            struct Varyings   { float4 positionCS : SV_POSITION; };

            // Uniform buffer — SRP Batcher compatible
            CBUFFER_START(UnityPerMaterial)
                half4 _Color;
            CBUFFER_END

            Varyings vert(Attributes input)
            {
                Varyings output;
                output.positionCS = TransformObjectToHClip(input.positionOS.xyz);
                return output;
            }

            half4 frag(Varyings input) : SV_Target
            {
                return _Color;  // uniform color from Material Inspector
            }
            ENDHLSL
        }
    }
}

CBUFFER / SRP Batcher: wrapping _Color in UnityPerMaterial constant buffer enables the SRP Batcher — GPU caches material data across draw calls. Without it the Batcher silently falls back to non-batched mode.

03 Shader Identity & Property Binding

Two tasks happen at the top of every shader file: establish the shader's identity (namespace path for Unity's database and Inspector menus) and declare Inspector-editable properties. Properties expose textures, colors, floats, and vectors to the Material Inspector — they drive runtime behavior by updating uniform buffers without triggering shader recompilation.

Shader "Custom/Pipeline/ExampleFlow"
{
    Properties
    {
        // Display Name       Type             Default
        _Color   ("Color",    Color)         = (1, 1, 1, 1)
        _MainTex ("Texture",  2D)            = "white" {}
        _Value   ("Strength", Range(0, 1))   = 0.5
        _VecProp ("Vector",   Vector)        = (0, 0, 0, 0)
        _IntVal  ("Count",    Integer)       = 4
    }
    // ... SubShader follows
}

No semicolons at the end of Properties lines in ShaderLab syntax. Semicolons are for HLSL code inside HLSLPROGRAM blocks only.

[Normal]          _BumpMap   ("Normal Map",      2D)    = "bump"  {}

[HDR] is critical in HDRP — without it the tonemapper converts emission values incorrectly. [Gamma] ensures float properties defined in sRGB space are converted properly when the project uses linear color space.

04 SubShader Setup — Tags & Queue

Every shader needs at least one SubShader. Unity selects the most compatible SubShader based on target hardware and active render pipeline. Tags are key–value pairs providing essential metadata: draw order, render type, and pipeline-specific behavior.

The render queue determines the order in which objects are drawn. The GPU sorts draw calls by these numerical ranges:

05 Blending Modes

Blending combines the fragment shader output (SrcValue) with the existing render target color (DstValue). The blend equation:

FinalColor = SrcFactor × SrcValue + DstFactor × DstValue

The default operation is Add. Alternative operations: Sub, RevSub, Min, Max (set via BlendOp).

Blend Factors Reference

FACTORRGB VALUEUSE CASE
One(1,1,1)Full color contribution — additive glow, emission
Zero(0,0,0)Eliminate the value entirely
SrcAlpha(A,A,A)Standard transparency — UI, glass
OneMinusSrcAlpha(1-A,1-A,1-A)Inverse alpha mask — soft particles
DstColor(R,G,B)Multiplicative — stained glass
SrcColor(R,G,B)Source color overlay
OneMinusDstColor(1-R,1-G,1-B)Soft light effects

Blend Presets

Pipeline Implementation

// Option A: Hardcoded in shader (BiRP)
SubShader {
    Tags { "Queue" = "Transparent" }
    Blend SrcAlpha OneMinusSrcAlpha
    ZWrite Off  // disable depth write for transparency
    Pass { CGPROGRAM /* ... */ ENDCG }
}

// Option B: Artist-controlled via Inspector (URP)
Properties {
    [Enum(UnityEngine.Rendering.BlendMode)] _SrcBlend ("Src Factor", Float) = 1
    [Enum(UnityEngine.Rendering.BlendMode)] _DstBlend ("Dst Factor", Float) = 0
}
SubShader {
    Tags { "RenderPipeline" = "UniversalPipeline" "Queue" = "Transparent" }
    Blend [_SrcBlend] [_DstBlend]
    ZWrite Off
    Pass { HLSLPROGRAM /* ... */ ENDHLSL }
}

Troubleshooting

ISSUEFIX

06 AlphaToMask & ColorMask

AlphaToMask

Forces the alpha channel to behave as a binary mask (0 or 1) instead of a gradient. Produces hard-edged transparency — foliage cutouts, portal effects, any scenario where crisp edges are required. Unlike blending, no Queue change is needed.

Shader "Custom/AlphaToMaskExample" {
    Properties { _MainTex ("Texture", 2D) = "white" {} }
    SubShader {
        Tags { "RenderType" = "Opaque" }
        AlphaToMask On  // converts alpha output → binary 0/1 mask
        Pass {
            CGPROGRAM
                // shader code here
            ENDCG
        }
    }
}

ColorMask

Restricts which color channels the GPU writes to the render target. Default is RGBA (all channels). Use for channel isolation, debugging, or stencil mask passes:

COMMANDWRITESUSE CASE
ColorMask RGBAAll channelsDefault behavior
ColorMask RGBColor only, no alphaOpaque pass optimization
ColorMask RRed onlyChannel isolation / debug
ColorMask AAlpha onlyDebug transparency values
ColorMask 0NothingStencil mask pass — invisible but writes stencil

// Different texture per face via SV_IsFrontFace:

Transparent Objects & Z-Buffer

  • Opaque: Render front-to-back — Z-Buffer correctly discards hidden pixels.
  • Transparent: Render back-to-front with ZWrite Off — prevents partially overlapped pixels from being incorrectly discarded.
  • Z-fighting: Occurs when two surfaces share identical depth values — GPU can't determine ordering, producing flickering. Fix: ZWrite Off, or Offset -1, -1.
Fig 07b: Co-planar geometry conflict resulting in Z-fighting artifacts.

Stencil Buffer

An 8-bit integer buffer (0–255) per pixel in the Frame Buffer. The GPU performs a Stencil Test before executing the fragment shader — acts as a pixel-level mask. Test equation:

if ( (StencilRef & ReadMask) [Comp] (StencilBuffer & ReadMask) )
    Accept pixel → proceed to ZTest → Fragment Shader
else
    Discard pixel

Stencil Comparison Functions

COMP VALUECONDITION
AlwaysAlways passes — used for writing the mask
EqualPasses if ref == buffer value
NotEqualPasses if ref ≠ buffer value — render everywhere except mask
Less / GreaterOrdered comparison
LEqual / GEqualInclusive ordered comparison
NeverAlways fails — pixel never draws

Practical Stencil Mask — Two Shader Pattern

// SHADER 1: Mask writer — invisible, writes stencil value 2
Shader "Custom/StencilMask" {
    SubShader {
        Tags { "Queue" = "Geometry-1" }  // process before default geometry
        ZWrite Off
        ColorMask 0  // invisible — no color output
        Stencil {
            Ref  2
            Comp Always   // always write
            Pass Replace  // set buffer value to Ref (2)
        }
        Pass { CGPROGRAM /* minimal */ ENDCG }
    }
}

// SHADER 2: Masked object — renders only where stencil ≠ 2
Shader "Custom/StencilObject" {
    Properties { _Color ("Color", Color) = (1,1,1,1) }
    SubShader {
        Tags { "Queue" = "Geometry" }
        ZWrite On
        Stencil {
            Ref  2
            Comp NotEqual  // skip pixels where mask wrote its value
            Pass Keep      // don't modify stencil buffer
        }
        Pass { CGPROGRAM
            #pragma vertex vert
            #pragma fragment frag
            fixed4 _Color;
            struct appdata { float4 vertex : POSITION; };
            struct v2f    { float4 pos    : SV_POSITION; };
            v2f   vert(appdata v) { v2f o; o.pos = UnityObjectToClipPos(v.vertex); return o; }
            fixed4 frag(v2f i) : SV_Target { return _Color; }
        ENDCG }
    }
}

08 CGPROGRAM / HLSLPROGRAM Reference

The programmable GPU stage begins here. Full-featured BiRP shader with all annotation layers:

Shader "Custom/FullReference"
{
    // ── 1. INSPECTOR INTERFACE ───────────────────────
    Properties
    {
        _MainTex ("Texture", 2D)    = "white" {}
        _Color   ("Color",   Color) = (1, 1, 1, 1)
    }

    // ── 2. RENDER PIPELINE CONFIG ────────────────────
    SubShader
    {
        Tags { "RenderType"="Opaque" }
        LOD 100

        // ── 3. GPU PROGRAM ────────────────────────────
        Pass
        {
            CGPROGRAM
            #pragma vertex vert      // declare vertex function
            #pragma fragment frag    // declare fragment function
            #pragma multi_compile_fog
            #include "UnityCG.cginc"

            // Global uniforms (outside CBUFFER for BiRP)
            sampler2D _MainTex;
            fixed4    _Color;

            // ── 4. DATA STRUCTURES ────────────────────
            struct appdata            // CPU → Vertex
            {
                float4 vertex : POSITION;
                float2 uv     : TEXCOORD0;
            };

            struct v2f                // Vertex → Fragment
            {
                float4 vertex : SV_POSITION;
                float2 uv     : TEXCOORD0;
                UNITY_FOG_COORDS(1)
            };

            // ── 5. VERTEX STAGE ───────────────────────
            v2f vert(appdata v)
            {
                v2f o;
                o.vertex = UnityObjectToClipPos(v.vertex);   // OS → ClipSpace
                o.uv     = TRANSFORM_TEX(v.uv, _MainTex);
                UNITY_TRANSFER_FOG(o, o.vertex);
                return o;
            }

            // ── 6. FRAGMENT STAGE ─────────────────────
            fixed4 frag(v2f i) : SV_Target
            {
                fixed4 col = tex2D(_MainTex, i.uv) * _Color;
                UNITY_APPLY_FOG(i.fogCoord, col);
                return col;
            }
            ENDCG
        }
    }
    FallBack "Diffuse"
}

09 Vertex & Fragment Stages — Data Structures

Structs are compound data type declarations grouping multiple elements. Two primary structs form the data contract between pipeline stages:

Δ1 // appdata / Attributes
Vertex input — CPU → Vertex Shader. Stores per-vertex object properties: position, normals, UVs, tangents, vertex colors. BiRP: appdata. URP: Attributes.
Δ2 // v2f / Varyings
Vertex output — Vertex Shader → Fragment Shader. Rasterized, interpolated values passed per pixel. BiRP: v2f. URP: Varyings.

Struct syntax — semantic connects the field to a hardware resource:

struct name
{
    vectorType fieldName : SEMANTIC;
};

Vertex Transform — UnityObjectToClipPos

The first operation in the vertex stage: transform object-space vertices to clip space. This multiplies the model matrix by the VP matrix:

$$\text{ClipPos} = \text{UNITY\_MATRIX\_VP} \cdot \text{unity\_ObjectToWorld} \cdot \text{float4(posOS, 1.0)}$$

Fragment Stage — Output Types

// BiRP (Cg): fixed4 return type — 11-bit precision
fixed4 frag(v2f i) : SV_Target { return fixed4(1,0,0,1); }

// URP/HDRP (HLSL): half4 return type — 16-bit precision
half4  frag(Varyings i) : SV_Target { return half4(1,0,0,1); }

fixed4 is Cg-only and compiles only in BiRP. Use half4 for URP/HDRP compatibility. SV_Target replaced the old COLOR semantic in D3D10+ — always use SV_Target in modern shaders.

[ DATA_TYPES // PRECISION & USE CASES ]

Texture Sampling — Two Patterns

// Pattern A: Modern HLSL (separate Texture2D + SamplerState)

SamplerState   sampler_MainTex;

    return _MainTex.Sample(sampler_MainTex, i.uv);
}

// Pattern B: Legacy macro (Cg / BiRP shorthand)
sampler2D _MainTex;

fixed4 frag(v2f i) : SV_Target {
    return tex2D(_MainTex, i.uv);  // equivalent to .Sample()
}
