Отлично. Мы возвращаемся к проверенной архитектуре **«Tag-Driven Deployment»**. Она максимально прозрачна: один тег — один неизменный срез кода. Это исключает хаос с кэшированием CDN.

Ниже — финальный инженерный регламент для твоей системы.

---

## 📘 ATLAS FRAMEWORK: РЕГЛАМЕНТ РЕЛИЗА И ВЕРСИОНИРОВАНИЯ

### 1. Этап разработки и коммита

Вносишь изменения в файлы модулей (`core.css`, `typography.css`, или файлы в `/components/`).

```bash
# 1. Добавляем измененные файлы
git add .

# 2. Фиксируем коммит (информативное сообщение)
git commit -m "feat: "

# 3. Отправляем в мастер (это "черновой" код, он еще не релиз)
git push origin master

```
git pull

git add .
git commit -m "feat: "
git push origin master
git tag v1.4.1
git push origin v1.4.1
---

### 2. Этап релизного деплоя (Создание тега)

Как только готов сделать релиз для блога, выполняешь:

```bash
# 4. Вешаем тег (инкрементируй версию, например v1.3.9)
git tag v1.3.9

# 5. Пушим тег на GitHub (этот шаг инициирует Purge в GitHub Actions)
git push origin v1.3.9

```

---

### 3. Интеграция в Wix

В своем HTML-виджете на Wix используй **строго версию тега**. Это гарантирует, что браузер пользователя получит именно твой проверенный релиз, а не «актуальный, но еще не протестированный» код из мастера.

```html
<link rel="stylesheet" href="https://cdn.jsdelivr.net/gh/pavelzosim/atlas-framework@v1.3.9/cdn/atlas-framework.css">

```

---

### 4. Как проверить актуальность (Хинты)

#### А) Проверка "какой код сейчас в теге"

Чтобы убедиться, что твой последний релиз успешно подхватился:

1. Зайди в браузер по ссылке: `https://cdn.jsdelivr.net/gh/pavelzosim/atlas-framework@v1.3.9/cdn/atlas-framework.css` (подставь свой тег).
2. Скролль в **самый низ** файла.
3. Ищи там метку сборки, которую мы вшили: `/* Build Stamp: v1.3.9-release */` (или аналогичную). Если видишь её — CDN отдает свежую версию.

#### Б) Как понять, почему Wix не видит изменений

Если ты запушил тег, но на сайте старые стили:

* **Инспектор сети:** Открой `F12` -> вкладка `Network`. Обнови страницу. Найди в списке `atlas-framework.css`.
* **Заголовок `x-cache`:** Если там написано `HIT` — файл взят из кэша. Если `MISS` — файл успешно скачан заново.
* **Принудительный Purge:** Если видишь `HIT` (старый файл), но уверен, что тег правильный — просто перейди по адресу:
`https://purge.jsdelivr.net/gh/pavelzosim/atlas-framework@v1.3.9/cdn/atlas-framework.css`
Это "пингует" сервер jsDelivr, чтобы он забыл старую версию.

### 💡 Инженерные советы

* **Не используй `@master` в продакшене:** В мастере может лежать «грязный» код в процессе отладки. Wix должен знать только о релизных версиях (`@v1.x.x`).
* **Логика тегов:** Если ты случайно запушил тег `v1.3.9` с ошибкой, ты **можешь** удалить его (`git tag -d v1.3.9` и `git push origin :refs/tags/v1.3.9`), поправить код и запушить тег снова. Но будь осторожен: это может вызвать кратковременное «двоение» кэша в CDN. Лучше просто выпускать `v1.4.0`.

Этот план понятен? Мы зафиксировали систему, которая не требует от тебя участия в «магии» сервера, а дает полный контроль через Git-теги.
